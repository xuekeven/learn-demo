# 排查解决问题（Help Fix Bug）

## 触发条件

在对话以以下任一方式表达排查/修复意图时，**主动按本 Skill 流程执行**：

- 以 `/help-fix` 开头（可带空格后接**具体问题描述**、**问题报告链接**、**项目产品链接**，可组合）
- 「看看这个问题/bug」「解决这个问题/不过」「排查一下」「看下这个报错」等**求助排查/修复**的表达（后接**具体问题描述**、**问题报告链接**、**项目产品链接**，可组合）

**产品链接**（`*` 表示可变片段）：`https://*.jd100.com/*`（子域名为占位，路径与查询串可变）。  
示例：`https://jdldoq.jd100.com/checkQuestion?subjectId=3&questionId=61599`

**问题链接**（`*` 表示可变片段）：

- 禅道 bug：`https://zentao.jiandan100.cn/zentao/bug-view-*.html`  
  示例：`https://zentao.jiandan100.cn/zentao/bug-view-59554.html`
- Jira bug：`https://jira.jiandan100.cn/jira/browse/JDRW-*`  
  示例：`https://jira.jiandan100.cn/jira/browse/JDRW-129337`
- 前端监控：`https://fe-monitor.jd100.com/easytech/issues/*?project=*`（`issues/` 后 issue id 与 `project` 参数可变）  
  示例：`https://fe-monitor.jd100.com/easytech/issues/24150?project=10`

若用户明确只要解释概念、不要做排查，则不要强行套本流程。

## 目标

在尽量少的来回中定位 bug 产生原因，总结生成 bug 报告，给出多种**可验证**的修复方案或明确阻塞点（需用户本机权限/密钥/业务数据时如实说明），并排查整个项目是否存在相同的问题。

## 工作流（按顺序）

### 1. 澄清问题陈述

用简短追问补齐（缺什么问什么，不要一次堆太多）：

1、 **问题分支**：确认bug发生在哪个分支，以便分析相对应的分支代码，以及修改代码时需要从该分支迁出新建分支（默认当前分支，跳过即用默认分支，用户输入则按用户输入为准，如果用户输入分支不存在，要求重新输入）
- **期望行为** vs **实际行为**
- **复现步骤**（含前置条件：分支、配置、账号角色）
- **首次出现时间/范围**（全员 vs 单机、是否与新提交/发布相关）
- **错误表现**：截图或原文、HTTP 状态码、堆栈顶部几行即可

均可跳过。

### 2. 收集证据（优先可机器验证）

- 终端/构建/测试失败输出、浏览器控制台与网络面板（若是前端）
- 相关日志字段、请求 ID、 correlation id（若有）
- **环境**：OS、运行时版本、依赖锁定文件是否变动

### 3. 形成假设并缩小范围

- 先区分：**数据/配置**、**逻辑**、**并发/时序**、**集成/网络**、**权限** 等类别
- 用代码搜索、断点阅读、最小复现实验缩小到**单文件或单函数**级别再改代码

### 4. 修复原则

- **最小改动**：只改解决问题所需代码，避免顺带重构
- **一致风格**：命名、错误处理、日志方式与周边代码一致
- **可回滚**：必要时说明风险与回滚方式（迁移、数据修复类）

### 5. 验证

- 运行与用户场景等价的命令或测试；无法运行时列出**用户侧验证步骤清单**
- 若有自动化测试，优先补充覆盖**本次缺陷场景**

## 与用户沟通方式

- 先说**结论或当前最可能原因**，再给**证据链**（日志行、代码引用）
- 引用现有代码时使用 Cursor 要求的代码引用格式（`start:end:path`）
- 阻塞时明确写出：**缺少什么信息**、**谁能提供**、**下一步**
