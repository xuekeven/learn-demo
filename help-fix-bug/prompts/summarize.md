# 总结 bug

## 用户交互

- 在分析与修复 bug 已完成后，询问用户是否输出一份结构化总结，便于工单回填、评审与留档。
  - 如果不需要，流程结束
  - 如果需要，按照下面的“文档模板”生成文档
  - 如果需要，用户告诉文档名以及保存在何处，按照用户要求新建并生成文档
  - 如果需要，用户未告诉文档名，默认按照分支名创建 markdown 文件 `fix-${YYMMDD}-${HHmm}.md` 作为文档名。
  - 如果需要，用户未告诉文档保存在何处，默认保存在当前工作项目下。

```
当前 bug 已修复！是否需要总结沉淀此 bug 为文档，便于工单回填、评审与留档？

如果需要，请告诉我文档如果命名以及保存在何处。
```

## 文档模板

按下面结构生成 Markdown 文档；无内容的节可写「无」或省略该节。

---

# Bug 修复报告：${bug_appear_baseline} · ${bug_appear_time}

## 概述

| 项目 | 内容 |
|------|------|
| 现象（一句话） | ${现象简述} |
| 严重度 / 影响面 | ${严重度，如"阻断主流程 / 部分用户受影响 / 偶现视觉问题"} |
| 代码基线 | `${bug_appear_baseline}` |
| 首次出现时间 | ${bug_appear_time} |
| 修复分支 | `fix-${YYMMDD}-${HHmm}` |

## 环境

- **客户端**：${浏览器/App WebView 及版本，若已知}
- **服务端 / 构建**：${与问题相关的环境，若已知}
- **信息来源**：${用户口述 / Jira 链接 / 禅道链接 / 前端监控链接 / 代码推断}

## 复现

**步骤**：

1. ${前置条件（登录态、数据、开关等）}
2. ${操作步骤}
3. ……

**期望**：${期望行为}

**实际**：${实际现象}

**频率**：${必现 / 偶现（触发条件：……）}

## 根因分析

**主因**：${一句话结论}

**证据链**：

- ${关键日志 / 堆栈 / 接口返回（使用 `startLine:endLine:path` 引用代码）}
- ${事实标注：来自工单 / 监控可核对}
- ${推断标注：根据代码路径推出}

**同类排查**：${仓库内是否存在相同反模式；若已扫描，写结论；若未扫，写建议后续项}

## 修复方案

**采用方案**：${方案名称及简述}

**变更摘要**：

| 文件 / 模块 | 变更说明 |
|-------------|----------|
| `${path/to/file}` | ${改动一两句话} |

**未本次修复的同类问题**：${列出或写"无"}

## 验证

**验证步骤**：

1. ${操作步骤}
2. ……
3. 预期结果：${……}

**回归注意点**：${相邻功能、边界数据、发布 / 配置依赖}

## 后续与风险

- **待办**：${需产品 / 后端 / 运维配合的事项，若无写"无"}
- **上线风险**：${说明}
- **回滚方式**：${回滚 commit / feature flag / 脚本逆操作等，若无写"无"}
- **工单摘要**（可直接粘贴到 Jira / 禅道「结论」字段）：

  > ${1～3 句，含根因 + 修复点 + 验证结论，脱敏处理敏感路径或参数}
