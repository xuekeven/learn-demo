# 修复 bug

前置：

- 在 `analyze.md` 阶段，用户已确认 bug 原因。
- **当前分支**应为 `create-branch.mjs` 创建的 `fix-`* 分支（或用户指定的等价修复分支）。

## 选择方案

### 提供方案

- 尽量给出 2～3 个可验证的解决方案并用表格对比，相关列：改动量、兼容性、性能、可维护性、风险等。  
- 对比维度可参考：老版本 **iOS / Android / Chromium**、弱网、首屏与交互性能、对公共组件的影响。  
- 默认推荐一个方案（通常为风险可控且可维护性较好的一条）并简短说明理由。

### 修改方案

- 如果用户对列出的方案不满意，重新思考解决方案。
- 如果用户对列出的某个方案有建议，思考用户的建议是否正确。如果正确，复述综合后的新方案；如果错误，告诉用户理由。

### 用户交互

用户确认选择了某个方案后再进入下一个“修改代码”步骤。

```
下面是我给出的解决方案：

${修复方案}

请你选择一个方案，我会根据你选择的方案直接上手修复代码。如果你对这些方案都不满意或者有疑问，可以告诉我。
```

```
下面是重新思考的解决方案：

${修复方案}

请你选择一个方案，我会根据你选择的方案直接上手修复代码。如果你对这些方案都不满意或者有疑问，可以告诉我。
```

```
这是综合你的意见后新的${某个方案}：

${新的某个方案}

如果可以，我会根据这个方案直接上手修复代码。
```

## 修改代码

### 原则

- **最小改动**：只改解决问题所需代码，不重构，不格式化代码，不影响 bug 之外的历史逻辑。  
- **一致风格**：命名、错误处理、日志、类型与原代码库一致；遵守项目既有 lint/格式化约定。  
- **可回滚**：涉及配置开关、接口契约、数据写入、迁移时，在回复中说明风险与回滚方式。

### 实现顺序

- 在最小复现路径上改**一处**主因；避免同时改多文件却未验证。
- 若有同类问题（分析阶段已列出），评估询问是本次一并修还是单独开项；一并修时仍保持 Diff 可评审。
- 补必要的防御性逻辑（空值、边界、错误分支），避免「修 A 引入 B」。
- 过程中，需要任何权限，及时向用户申请；出现任何阻塞，及时向用户报告。

### 用户交互

- 修改完成代码后，要求用户手动检查 bug 是否已修复，并分析用户的回答
  - 如果已经修复，进入下一流程
  - 方案方向正确，只是还没有实现预期，继续思考修改代码
  - 确认目前的解决方案行不通，或者有新的解决方案，经过用户确认后，撤回修改的代码，回到“选择方案”步骤
  - 确认了 bug 原因不准确，或者新的 bug 原因，经过用户确认后，撤回修改的代码，回到之前流程的“综合分析”步骤

输出原则：

- 简述**改了什么文件/行为**、为何能对症。  
- 对关键逻辑用**代码引用**标出。  
- **阻塞**（需设计拍板、后端接口、合规等）单独成段，不写进代码里糊弄过去。

```
修改代码已完成，请手动检查 bug 是否已修复。有任何问题都直接告诉我。
```
